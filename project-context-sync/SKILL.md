# Project Context Sync

Automatically analyze a project's structure, detect its tech stack and dependencies, then update CLAUDE.md and generate a comprehensive project report.

## When to Use

Use this skill when:
- User explicitly asks to "sync project context" or "update CLAUDE.md"
- User switches to a new project and wants to understand it
- User asks "what's this project about?" or "what tech stack does this use?"
- User wants to check dependencies or project configuration
- User mentions analyzing, scanning, or understanding the current codebase

## What This Skill Does

1. **Scans the project** - Analyzes file structure, package files, config files
2. **Detects tech stack** - Identifies languages, frameworks, tools, and dependencies
3. **Updates CLAUDE.md** - Creates or updates the project's CLAUDE.md with essential context
4. **Generates report** - Creates a detailed analysis report for reference

## Workflow

### Step 1: Identify Project Root

First, determine the project root directory:
- If user specifies a path, use that
- Otherwise, use the current working directory
- Look for indicators: `.git/`, `package.json`, `requirements.txt`, `go.mod`, etc.

### Step 2: Scan Project Structure

Analyze the project by examining:

**Package/Dependency Files** (check in this order):
- `package.json` / `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` - Node.js
- `requirements.txt` / `Pipfile` / `pyproject.toml` / `poetry.lock` - Python
- `go.mod` / `go.sum` - Go
- `Cargo.toml` / `Cargo.lock` - Rust
- `pom.xml` / `build.gradle` - Java
- `Gemfile` / `Gemfile.lock` - Ruby
- `composer.json` - PHP

**Configuration Files**:
- `.env.example` / `.env.template` - Environment variables
- `tsconfig.json` / `jsconfig.json` - TypeScript/JavaScript config
- `.eslintrc.*` / `.prettierrc.*` - Linting/formatting
- `docker-compose.yml` / `Dockerfile` - Docker setup
- `vercel.json` / `netlify.toml` - Deployment config
- `.github/workflows/*.yml` - CI/CD pipelines

**Documentation**:
- `README.md` - Project description
- `CONTRIBUTING.md` - Contribution guidelines
- `docs/` directory - Additional documentation
- Existing `CLAUDE.md` - Previous context

**Code Structure**:
- Main source directories (`src/`, `lib/`, `app/`, etc.)
- Test directories (`test/`, `tests/`, `__tests__/`, `spec/`)
- Build output directories (`dist/`, `build/`, `out/`)

### Step 3: Extract Key Information

From the scanned files, extract:

**Tech Stack**:
- Primary language(s) and versions
- Frameworks (React, Next.js, Django, Express, etc.)
- Key libraries and their purposes
- Build tools (Webpack, Vite, esbuild, etc.)
- Testing frameworks (Jest, Pytest, Go test, etc.)

**Project Metadata**:
- Project name and description (from package.json, README, etc.)
- Version number
- Author/maintainer information
- License

**Dependencies**:
- Production dependencies (count and notable ones)
- Development dependencies (count and notable ones)
- Peer dependencies or optional dependencies

**Configuration**:
- Environment variables needed (from .env.example)
- API endpoints or external services used
- Database connections (if evident)

**Scripts/Commands**:
- Available npm/yarn scripts or make targets
- Common development commands (dev, build, test, lint)
- Deployment commands

### Step 4: Update CLAUDE.md

Create or update `CLAUDE.md` in the project root with this structure:

```markdown
# [Project Name]

> Auto-generated by project-context-sync on [date]

## Project Overview

[Brief description from README or package.json]

## Tech Stack

**Primary Language**: [language + version]
**Framework**: [main framework]
**Key Libraries**:
- [library 1] - [purpose]
- [library 2] - [purpose]
- [library 3] - [purpose]

## Project Structure

```
[key directories and their purposes]
```

## Development

**Setup**:
```bash
[installation commands]
```

**Common Commands**:
- `[dev command]` - Start development server
- `[build command]` - Build for production
- `[test command]` - Run tests
- `[lint command]` - Lint code

**Environment Variables**:
[list from .env.example if exists]

## Dependencies

- **Production**: [count] packages
- **Development**: [count] packages

Notable dependencies: [list 5-10 most important ones]

## Notes

[Any special considerations, gotchas, or important context]
```

**Important Guidelines**:
- Keep it concise - aim for 100-200 lines max
- Focus on information that helps Claude understand the project quickly
- Include file paths for key configuration files
- Highlight any unusual patterns or project-specific conventions
- If CLAUDE.md already exists, preserve user-written sections and only update auto-generated parts

### Step 5: Generate Analysis Report

Create a detailed report at `docs/project-analysis-[date].md` with:

**Full Dependency List**:
- All production dependencies with versions
- All dev dependencies with versions
- Dependency tree depth and complexity metrics

**File Structure Analysis**:
- Total file count by type
- Largest files and directories
- Code organization patterns

**Configuration Details**:
- All detected config files and their purposes
- Environment variable requirements
- API integrations and external services

**Recommendations** (optional):
- Outdated dependencies that should be updated
- Missing documentation
- Potential security issues (if evident)
- Suggested improvements to project structure

### Step 6: Present Results

Show the user:
1. A summary of what was detected
2. Path to the updated CLAUDE.md
3. Path to the detailed analysis report
4. Any warnings or issues found

Example output:
```
✅ Project context synced successfully!

📊 Detected:
- Language: TypeScript 5.3
- Framework: Next.js 14.1
- 47 production dependencies, 23 dev dependencies

📝 Updated: /path/to/project/CLAUDE.md
📄 Report: /path/to/project/docs/project-analysis-2026-03-04.md

⚠️ Notes:
- Found .env.example with 8 required variables
- No tests directory detected
```

## Edge Cases

**No package files found**:
- Still analyze file structure and create basic CLAUDE.md
- Note in the report that dependency detection was not possible

**Multiple tech stacks** (e.g., monorepo):
- Detect all stacks and list them separately
- Ask user which one to focus on for CLAUDE.md

**Existing CLAUDE.md with user content**:
- Preserve sections that don't look auto-generated
- Add a comment marking auto-generated sections
- Offer to show diff before overwriting

**Large projects** (1000+ files):
- Focus on key directories only
- Sample files rather than reading everything
- Warn user that analysis may take time

## Tips for Accuracy

- Read package files completely to get accurate dependency lists
- Check multiple sources for project description (README, package.json, etc.)
- Look for framework-specific patterns (e.g., `next.config.js` indicates Next.js)
- Detect monorepo tools (Turborepo, Nx, Lerna) and handle accordingly
- Identify CI/CD setup from `.github/workflows/` or similar
